## ADR-001: FSRS (Free Spaced Repetition Scheduler)

### Статус
**Принято**. FSRS выбран вместо SM-2.

### Контекст
SM-2 (SuperMemo 2) — классический алгоритм интервальных повторений, но он имеет недостатки:
- Использует фиксированный ease factor, который не адаптируется к пользователю
- Не учитывает «сложность» карточки как независимый параметр
- Требует больше повторений для достижения того же уровня запоминания

FSRS (версия 5) использует модель памяти на основе трёхкомпонентной теории:
1. **Stability (S)** — стабильность памяти (в днях). Растёт с каждым успешным повторением.
2. **Difficulty (D)** — врождённая сложность материала (0..1). Адаптируется после каждого ответа.
3. **Retrievability (R)** — вероятность вспоминания в момент повторения. Вычисляется как `R = exp(ln(0.9) * elapsed_days / S)`.

### Параметры FSRS v5

FSRS использует 17 обучаемых параметров (w0..w16), которые калибруются на исторических данных пользователя:

| Параметр | Значение по умолчанию | Описание |
|----------|----------------------|----------|
| w0 | 0.4 | Начальная стабильность после первого обучения (Again) |
| w1 | 0.6 | Начальная стабильность после первого обучения (Hard) |
| w2 | 2.4 | Начальная стабильность после первого обучения (Good) |
| w3 | 5.8 | Начальная стабильность после первого обучения (Easy) |
| w4 | 4.93 | Коэффициент роста стабильности (Again) |
| w5 | 0.94 | Коэффициент роста стабильности (Hard) |
| w6 | 0.86 | Коэффициент роста стабильности (Good) |
| w7 | 0.01 | Коэффициент роста стабильности (Easy) |
| w8 | 1.49 | Штраф за неудачу (Again → влияет на difficulty) |
| w9 | 0.14 | Штраф за Hard |
| w10 | 0.94 | Бонус за Good (снижает difficulty) |
| w11 | 2.18 | Бонус за Easy |
| w12 | 0.05 | Смещение difficulty |
| w13 | 0.34 | Вес stability в формуле difficulty |
| w14 | 1.26 | Множитель для retrieval probability |
| w15 | 0.29 | Смещение для retrieval probability |
| w16 | 2.61 | Множитель для интервала |

**Глобальные константы:**
- `DECAY = -0.5` — коэффициент затухания difficulty
- `FACTOR = 0.9` — целевая retrievability (= 90% вероятность вспоминания)
- `MAXIMUM_INTERVAL = 36500` — максимальный интервал (100 лет)

### Рейтинги ответов (совместимы с Anki)

| Рейтинг | Значение | Описание |
|---------|---------|----------|
| Again | 1 | Совсем не помню — сброс в learning |
| Hard | 2 | Вспомнил с трудом |
| Good | 3 | Вспомнил нормально |
| Easy | 4 | Вспомнил легко |

### Текущий статус реализации

Файл `apps/server/src/modules/sessions/srs.ts` содержит **упрощённую заглушку** FSRS с табличными коэффициентами (difficultyDelta, stabilityMultiplier, intervalMultiplier). Это не полная реализация FSRS v5 — она не использует параметры w0..w16 и формулу retrievability.

**План:** Полная реализация FSRS v5 будет выполнена на Этапе 1 (SRS-ядро). Заглушка достаточно точна для разработки и тестирования интерфейса.

## ADR-002: Выбор ORM — Prisma вместо Drizzle

### Статус
**Принято** (отклонение от плана PLAN.md п.0.1.7).

### Контекст
Изначальный план (`plan/PLAN.md`, пункт 0.1.7) предполагал использование **Drizzle ORM**. Однако на момент начала разработки репозиторий уже содержал полную реализацию на **Prisma**:
- Полная схема БД (`apps/server/prisma/schema.prisma`) — 8 моделей + 1 enum
- 4 сервисных модуля (auth, words, sessions, stats), полностью использующих PrismaClient
- Настроенные скрипты миграции (`db:migrate`, `db:push`, `db:generate`, `db:seed`)
- Многоэтапный Dockerfile, включающий `prisma generate` на стадии production

### Решение
Оставить **Prisma** как основной ORM.

### Обоснование
1. **Нецелесообразность переписывания.** Замена на Drizzle потребовала бы переписывания ~400+ строк серверного кода в 6 файлах без добавления новой функциональности.
2. **Prisma полностью покрывает требования.** Все таблицы из PLAN.md (Users, Words, Decks, Deck_Words, User_Word_Progress) реализованы в Prisma-схеме.
3. **Зрелость экосистемы.** Prisma имеет встроенную поддержку миграций, генерацию типов, студийный UI — всё, что нужно для Stage 0.
4. **Команда знакома с Prisma.** Стек `Prisma + Fastify + Zod` хорошо задокументирован и широко используется.

### Последствия
- Путь к seed-данным: `apps/server/prisma/seeds/` (вместо `packages/backend/src/db/seeds/`)
- Миграции: `apps/server/prisma/migrations/` (вместо Drizzle Kit)
- Типы генерируются через `prisma generate`, а не выводятся из Drizzle-схемы
- Zod-схемы в `packages/shared/src/schemas/` остаются независимым источником истины для API-контрактов

## ADR-003: Структура монорепозитория — apps/ вместо packages/

### Статус
**Принято** (отклонение от плана PLAN.md п.0.3.1).

### Контекст
План предполагал структуру `packages/backend`, `packages/frontend`, `packages/shared`. Фактически используется:
```
apps/server/    → @hanzi/server (Fastify backend)
apps/web/       → @hanzi/web (React + Vite frontend)
packages/shared/ → @hanzi/shared (Zod схемы + типы)
```

### Решение
Сохранить текущую структуру. Разделение `apps/` (запускаемые приложения) и `packages/` (библиотеки) — стандартная практика для Turborepo.

## ADR-004: Миграции — версионирование вместо db push

### Статус
**Принято**.

### Контекст
До Stage 0 схема БД синхронизировалась через `prisma db push` (прямая синхронизация без истории). Это удобно для прототипирования, но неприемлемо для:
- Командной разработки (коллеги не видят историю изменений)
- Production-деплоя (нет контроля над применением миграций)
- Отката изменений

### Решение
Использовать `prisma migrate dev` для создания версионированных миграций. Скрипт `db:push` сохраняется для быстрого прототипирования.

### SQL миграции хранятся в
`apps/server/prisma/migrations/<timestamp>_<name>/migration.sql`
