### Этап 0: Подготовка и проектирование (Архитектурный фундамент)

До написания первой строчки кода нужно зафиксировать контракты данных.

1. **Проектирование схемы БД (PostgreSQL):**
    
    - `Users`: id, email, password_hash, xp, current_streak, last_active_date.
        
    - `Words`: id, character, pinyin, translation, hsk_level (чтобы сразу залить базу для HSK 1 и HSK 2 для твоих текущих тренировок), audio_url, mnemonic.
        
    - `Decks`: id, name, description, is_system_deck (например, "Еда", "HSK 2").
        
    - `User_Word_Progress` (Сердце SRS): user_id, word_id, state (new, learning, review, graduated), due_date, stability, difficulty, reps.
        
    - _Скептичная заметка:_ В манифесте упомянут алгоритм SM-2 (как в Anki), но в 2026 году лучше сразу брать **FSRS (Free Spaced Repetition Scheduler)**. Он точнее и требует меньше итераций для запоминания.
        
2. **Проектирование API-контрактов:**
    
    - Описать OpenAPI/Swagger схему для всех эндпоинтов, особенно `/sync` для обмена дельтами.
        
3. **Настройка репозитория и CI/CD:**
    
    - Монорепозиторий (Turborepo или просто yarn workspaces), чтобы шарить типы Zod между фронтом (PWA) и бэкендом (Fastify).
        
    - Github Actions для прогона линтеров и сборки Docker-образов.
        

### Этап 1: Бэкенд и Инфраструктура (Базовая логика)

Настроить серверную часть и поднять базу. Учитывая твой опыт администрирования серверов и работы с доменами, настройка TLS 1.3 и деплой Docker-контейнеров на AWS или Railway пройдет как по маслу.

1. **Инициализация Fastify:**
    
    - Подключение PostgreSQL (через Prisma или Drizzle ORM — Drizzle сейчас работает быстрее и весит меньше).
        
    - Настройка Redis для кэширования и Rate Limiting (закрываем API от брутфорса).
        
2. **Модуль Авторизации:**
    
    - Реализация JWT (Access + Refresh).
        
    - Хранение Refresh-токена в HttpOnly cookie, Access-токена — в памяти клиента (или тоже в cookie).
        
3. **CRUD для слов и колод:**
    
    - Эндпоинты для получения списков слов.
        
    - Парсер/скрипт для первоначальной заливки словаря (можно спарсить готовые CSV списки HSK).
        
4. **Реализация SRS-движка на сервере:**
    
    - Логика пересчета `due_date` при получении ответов (_Не помню, Помню, Легко_).
        
    - Эндпоинт генерации сессии: выдает N карточек, у которых `due_date <= now()`, плюс микс из новых слов.
        

### Этап 2: Фронтенд — Основа и UI-кит (Веб-версия / PWA)

Так как приоритет сейчас на веб-приложение, логично использовать Expo (React Native for Web) или чистый React (Vite) + PWA плагины.

1. **Инициализация приложения:**
    
    - Настройка React + Zustand (стейт-менеджер) + React Query (запросы).
        
    - Настройка PWA (Service Worker, `manifest.json`, чтобы приложение можно было "установить" на телефон из браузера).
        
2. **Верстка UI-кита (по прототипу):**
    
    - Реализация дизайн-системы: темная тема (фон `#0C0E16`, карточки `#141820`).
        
    - **Цветовое кодирование тонов (Критически важно):** Написать утилиту, которая парсит пиньинь (например, `xǐ huān`) и оборачивает слоги в нужные цвета (1 - синий, 2 - зеленый, 3 - желтый/оранжевый, 4 - красный, 0 - серый).
        
3. **Навигация и Лейаут:**
    
    - Боковое/нижнее меню (Главная, Учить, Слова, Итоги).
        
    - Роутинг (React Router).
        

### Этап 3: Ядро приложения — Режим тренировки

Это то, ради чего все затевается. Интерфейс не должен лагать, анимации должны быть в 60 FPS.

1. **Компонент Карточки (Flashcard):**
    
    - Верстка огромного иероглифа на лицевой стороне.
        
    - 3D-трансформация (CSS `transform: rotateY(180deg)`) для плавного переворота.
        
2. **Игровой цикл (Core Loop):**
    
    - Загрузка сессии (массив слов) через React Query.
        
    - Отображение карточки -> Переворот -> Выбор оценки (Кнопки "Не помню", "Помню").
        
    - Локальный стейт Zustand: сдвиг прогресс-бара, переход к следующей карточке.
        
3. **Интеграция аудио (Google TTS):**
    
    - При перевороте карточки должен автоматически проигрываться звук. Аудиофайлы лучше не генерировать на лету каждый раз, а стягивать с сервера (где бэкенд один раз сгенерировал их через Google TTS и положил в S3/базу).
        
4. **Анимация успеха:**
    
    - После завершения колоды — легкое конфетти (библиотека `canvas-confetti`) и экран итогов сессии.
        

### Этап 4: Экраны Библиотеки и Статистики

1. **Главный дашборд:**
    
    - Сборка данных для "Полосы огня" (streak) и количества слов на сегодня.
        
2. **Библиотека слов:**
    
    - Отображение сетки слов (wgrid из прототипа) с бейджами статуса.
        
    - Реализация поиска с дебаунсом (debounce) по иероглифу, пиньиню и русскому.
        
3. **Статистика (Heatmap):**
    
    - Реализация календаря активности (похоже на GitHub contributions).
        

### Этап 5: Синхронизация и Офлайн (Самая хардовая часть)

Делать сразу офлайн-first архитектуру очень дорого. Я рекомендую сначала запустить онлайн-версию, а офлайн прикручивать на втором этапе.

1. **Внедрение WatermelonDB (или RxDB для веба):**
    
    - Замена локального стейта Zustand (в части хранения слов) на локальную SQLite БД.
        
2. **Дельта-синхронизация:**
    
    - Каждое действие юзера (ответ на карточку) пишется в локальную БД с флагом `is_synced = false` и таймстемпом.
        
    - При появлении сети фронтенд пушит массив изменений на эндпоинт `/sync`.
        
    - Бэкенд разруливает конфликты (правило: кто ответил позже, тот и прав) и отдает фронту обновленные/новые слова, которые нужно скачать.
        
3. **Кэширование аудио:**
    
    - Service Worker должен скачивать mp3-файлы текущей изучаемой колоды в CacheStorage браузера.
        

### Этап 6: Полировка, Уведомления и Монетизация

1. **Push-уведомления (Firebase):**
    
    - Настройка FCM для PWA (работает на Android и iOS 16.4+).
        
    - Cron-джоба на сервере Fastify, которая раз в час проверяет базу и шлет пуш: "Пора повторить 24 слова".
        
2. **Рукописный ввод (Фича для продвинутых):**

3. **Интеграция Stripe (если планируется Pro-тариф):**
    
    - Вебхуки на сервере для выдачи премиума.
